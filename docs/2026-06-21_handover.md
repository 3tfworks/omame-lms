## 2. アフィリエイト報酬率システム

### 2.1 アーキテクチャ（前回踏襲）

**DB スキーマ**:
- `campaign_periods` テーブル: 期間ベースの報酬率管理
  - 列: `start_at` / `end_at` (timestamptz) / `reward_rate` (1-100) / `is_active` / `name`
  - EXCLUDE 制約: `tstzrange(start_at, end_at, '[)') WITH &&` WHERE `is_active` で重複防止
  - btree_gist 拡張使用
- `system_settings` テーブル: 通常報酬率の default 値（JSON）

**共通関数**:
- `src/lib/affiliateRate.ts` の `getAffiliateRewardRate(at?, client?)`
  - 引数 `at`: 報酬率の判定基準日時（指定なしなら現在時刻）
  - 引数 `client`: Supabase クライアント（指定なしなら service-role）

**3経路統一**:
1. Stripe webhook（`session.created` の `created` フィールドを `at` に渡す）
2. Kaihipay webhook（受信時刻を `at` に渡す）
3. invite redeem（現在時刻を `at` に渡す）

### 2.2 referrer_id 解決経路（フェーズ1 で改善）

`/api/checkout/stripe` での解決優先順位:

```
1. body.referrerId     … 招待フロー /invite/[userId]/thanks が明示指定
2. cookie referrer_id  … LP の ?ref=xxx 経路（proxy.ts が cookie 発行、server-side 読み取り）
3. なし                … metadata.referrer_id 未設定（webhook の email フォールバックに委譲）
```

**フェーズ1 で解消した silent attribution loss**:
- 以前は proxy が `?ref=xxx` から cookie を発行していても、`/api/checkout/stripe` が cookie を読まなかった
- LP `?ref=xxx` 経由の購入は metadata.referrer_id が未設定で、報酬は webhook の email フォールバック頼みだった
- フェーズ1 commit `6451e27` で server-side cookie 読み取りを実装し、`?ref=xxx` → cookie → checkout → webhook の一気通貫が成立

### 2.3 サロン経由の報酬対象外（フェーズ2 で新設）

`/api/webhooks/stripe` の `handleCheckoutCompleted` で:
- `session.metadata?.price_type === "salon"` の場合、**アフィリエイト報酬ブロック全体をスキップ**
- スキップ対象: `referrer_id` 経路の計算、`email → invite_leads` 逆引きフォールバック、`affiliate_rewards` への INSERT
- 受講登録 / Magic Link / メール送信などの受講者向け処理は通常実行
- ローカル検証で「salon + cookie referrer 両立」ケース（E）も実証済み

### 2.4 運用手順（管理画面）

`/admin/campaign` でキャンペーン期間の管理:
- タイムライン表示で現行・予定・過去のキャンペーンを一覧
- 追加・編集・停止（`is_active=false`）・削除が可能
- JST 入力 → timestamptz 保存（自動変換）

`/admin/affiliate` で通常報酬率の管理:
- 単一入力欄で default を設定

### 2.5 既知の運用ルール

- 期間重複は許可しない（DB 制約で保証）
- 緊急停止は `is_active=false` で実現（削除ではない）
- 価格表示（`product_pricing`）とは独立（販売価格と報酬率は別管理）
- **サロン経由（`/ja/lp-salon`）は常に報酬対象外**

### 3.2 priceType 伝搬経路（フェーズ2 で新設）

```
/ja/lp-salon (lp-salon/page.tsx, noindex, force-dynamic)
        │  <LpV2Page params={params} priceType="salon" />
        ▼
LpV2Page (lp-v2/page.tsx)  priceType を以下へ伝搬:
   - §1 Section01Hero(priceType)      → salon 時 eyebrow を "For Salon Members"
   - §11 Section11Price(priceType)    → salon バッジ + ¥9,800
   - §14 Section14FinalCta(priceType) → 同上
   - 各 CtaButton(priceType)
        ▼
CtaButton onClick → fetch POST /api/checkout/stripe
        body: { priceType: priceType ?? "general" }
        ▼
/api/checkout/stripe: priceType 検証 → getProductPricing(priceType) → salon Price ID
        → Stripe Session metadata.price_type = "salon"
        ▼
/api/webhooks/stripe: metadata.price_type === "salon" の場合、報酬ブロック全スキップ
```

### 3.3 重要な設計判断: PUT passthrough

`/admin/pricing` の PUT は `value`（JSON 全体）を置換するため、フェーズ2 で **現行の `salonPrice` / `salonStripePriceId` を SELECT して持ち越す（passthrough）** ロジックを追加（commit `6431fdc`）。

これが無いと、管理画面で general 価格を保存するたびに salon 2キーがサイレントに消失していた。販売不能事故の防止策。

### 3.4 fail-closed 設計

`getProductPricing("salon")` で salon Price ID が DB にも env（`STRIPE_PRICE_ID_SALON`・現状未設定）にも無い場合:
- 空 ID を返してログ出力
- `/api/checkout/stripe` が 500（Server misconfiguration）で安全停止
- 誤って general Price で salon 価格を売る事故を防止

### 3.5 DB の test/LIVE 移行の経緯

フェーズ1 着手前、DB の `stripePriceId` が **test mode Price** だった（過去のテスト時の残置）。LIVE キーで `resource_missing` になり Checkout が 500 で落ちる状態。SQL 直接更新で LIVE Price（`price_1ThrC2...`）へ修正済み。

## 6. 重要な設計判断と根拠

### 6.1 マーケティング判断（前回踏襲）

#### LP は「変化 → 証拠 → 理論」の順
購入者は理論を知りたいのではなく「変われるのか」を知りたい。読者心理は `本当か → 変われるか → 変わった人いる → 何が変わった → その正体は` の順。

#### ただし、お豆奏法は理屈派が買う商品
購入者層（ピアノ講師・指導者・音大卒）は理屈で納得しないと買わない。よって `変化 → 証拠 → 理解 → 納得 → 購入` の流れ。

#### サロン版での FV eyebrow 置換判断（フェーズ2）
- 一般版: `A New Way of Playing`（ブランドコンセプト宣言）
- サロン版: `For Salon Members` に**置換**（限定配布の文脈では「サロン専用」が第一印象として最も重要）
- 重複併記ではなく置換で認知負荷を下げる

#### サロン LP の noindex
- サロンメンバー限定のため検索エンジンに載らない
- robots: `noindex, nofollow`
- URL 漏洩リスクは「サロン少数で実害限定的」として許容（ランダムスラッグ等の認証は採用せず）

### 6.2 技術判断

#### Next/Image の正しい使い分け（前回踏襲）
- 全面背景（FV など LCP 対象）: `fill` + `priority` + 親に `position: relative`
- インライン画像: `width` / `height` + `loading="lazy"` + `decoding="async"`
- `fill` と `width/height` は両立しない、`priority` と `loading` は併用不可

#### SP 最適化は SectionShell に集約（前回踏襲）
共通の SectionShell で `px-4 md:px-5` を実装。保守性向上。

#### lp-v2 決済完結化の判断（フェーズ1）
- 引き継ぎ書 §8.3「LP v2 を本番販売 URL として採用するか」を「採用」と確定
- offer-demo 経由をやめ、`/ja/lp` の CTA から直接 Stripe Checkout へ
- offer-demo は凍結（削除せず、安定運用後に撤去判断）

#### referrer_id cookie 解決の server-side 実装（フェーズ1）
- proxy で発行された httpOnly cookie を `/api/checkout/stripe` が server-side で読み取り
- 解決優先順位: body > cookie > none
- 既存の招待フロー（明示 body.referrerId）は最優先で動作維持

#### サロン版の実装方式: 共通化（複製ではなく）（フェーズ2）
- 14セクションある大規模 LP を複製すると保守性が劇的に悪化
- 影響を受けるのは 4-5 ファイルのみ
- priceType prop 伝搬で実現、`/ja/lp-salon` は薄ラッパー

#### system_settings JSON の拡張方針: フラット拡張（フェーズ2）
- 既存スキーマを破壊しない
- `getProductPricing()` の戻り値構造を維持（後方互換）
- 新規呼び出しのみ `getProductPricing("salon")` で分岐

#### PUT passthrough によるデータ保全（フェーズ2）
- `/admin/pricing` PUT が salon 2キーをサイレント消失させるリスクを発見
- 「既存挙動維持」の例外として、フェーズ2 の必須改修扱い
- 管理画面 UI は変更せず、内部ロジックで salon キーを保持

#### Stripe Product は単一、Price のみ追加（フェーズ2）
- 「お豆奏法 基礎講座」は1つの Product
- 一般 ¥29,800 と サロン ¥9,800 は同 Product の異なる Price バリエーション
- レポート集計が一元化される、商品情報の二重管理を防げる

1. **金銭リスク**（報酬率・価格・決済）

4. **UX**（SP 読み進めやすさ・購入導線）
5. **技術的整合性**（型エラー・lint・CLS）

### 7.4 フェーズ1/2 で確立した堅牢パターン

- **コミット前確認の徹底**: `.env.local` 等の除外、対象ファイルのみのステージ、検証用残骸の除去
- **DB 変更前のバックアップ SELECT**: フェーズ1/2 とも UPDATE 前に SELECT で現値を保存
- **fail-closed 設計**: 価格 ID 欠落時に誤った Price で売る事故を防ぐ
- **passthrough によるデータ保全**: スキーマ拡張時に既存キーを破壊しない
- **軽量検証の活用**: Stripe Checkout の Session 作成のみで離脱、決済確定なしで metadata を実測
