# ギフトクーポン運用マニュアル

**初版**: 2026-06-25
**対象機能**: Stripe Checkout のプロモーションコード入力欄(¥0 配布対応)
**運用者**: 甲斐 / えりな先生

---

## 0. 概要

お豆奏法の有料講座(現状は基礎講座 ¥29,800)を、特定の招待者に対して **¥0 で配布** するための仕組み。Stripe のクーポン + プロモーションコード機能を活用し、コードを入力した購入者だけが ¥0 でアカウント作成できる。

### 0.1 動作の流れ

```
1. 招待者が LP(/ja/lp-v2)から購入フローに入る
2. Stripe Checkout 画面で「プロモーションコード」欄に
   ERINA-GIFT を入力
3. 金額が ¥29,800 → -¥29,800(100% 割引) → 合計 ¥0 に変化
4. メールアドレスを入力 → 「注文を完了」をクリック
5. Stripe で ¥0 取引が成立(課金は発生しないが取引履歴は残る)
6. Webhook 経由で auth.users / public.users 作成
7. ユーザーのメールに Supabase Magic Link が送信される
8. リンクをクリックして LMS にログイン → 動画視聴可能
```

### 0.2 ¥0 取引時の挙動(重要)

- `session.amount_total === 0` のため、アフィリエイト報酬計算は **自動でスキップ** される
- これは現在の報酬計算ロジック(`amount_total` を基に算出)の自然な帰結
- 紹介経由でギフト購入された場合も報酬は発生しない(意図的・正しい挙動)

### 0.3 実装の要点

- コード変更箇所: `src/app/api/checkout/stripe/route.ts:68` に `allow_promotion_codes: true` を追加(コミット `e590e47`)
- それ以外のコード変更なし(Stripe 標準機能で完結)

## 3. 新規クーポンの発行手順

ERINA-GIFT 以外の用途(別キャンペーン・別商品)で新規クーポンを発行する場合の手順。

### 3.1 Stripe Dashboard でクーポン作成

```
Stripe Dashboard → 商品カタログ → クーポン → クーポンを作成
```

入力項目:

| 項目 | 説明 | 例 |
|---|---|---|
| 名前 | Stripe 管理用の名称 | 夏のキャンペーン 30%OFF |
| 種別 | 「割引率(%)」or「割引額(円)」 | 30%(¥0 配布なら 100%) |
| 期間 | 「永続」or「複数月」or「1回」 | 1回(買い切り商品用) |
| 適用対象 | 「全商品」or「特定商品」 | 特定商品(おうちで学べる…)を推奨 |
| 引き換え上限 | 全体での最大引き換え回数 | 50(目安) |
| 有効期限 | クーポンが利用可能な期限 | 2026-08-31 |

### 3.2 プロモーションコードを発行

クーポン作成後、そのクーポンに対して「プロモーションコード」を発行:

```
作成したクーポンの詳細ページ
  → 「プロモーションコードを追加」
  → コード文字列を入力(例: SUMMER2026)
  → 1ユーザーあたりの利用回数(通常 1)
  → 最大引き換え回数(クーポン全体の上限とは別に設定可)
  → 有効期限
  → 保存
```

### 3.3 コードを共有

発行したプロモーションコードを、対象の購入者に共有(LINE / メール / SNS など)。

### 3.4 コード側の対応(基本不要)

`allow_promotion_codes: true` は既に有効化済みのため、新しいクーポンを Stripe 側で発行するだけで自動的に入力欄が動作する。コードのデプロイは不要。

## 4. トラブルシューティング

### 4.1 「プロモーションコードが無効です」と表示される

**原因の候補**:
- コードのスペル違い(大文字小文字は厳密に区別される)
- 引き換え上限に達している
- 有効期限切れ
- そのコードが「アクティブ」になっていない
- 商品の対象外(別商品の購入時にコードを適用しようとした)

**確認方法**:
```
Stripe Dashboard → 商品カタログ → クーポン
  → 該当プロモーションコードの状態を確認
  → 引き換え回数 / 有効期限 / 適用対象商品
```

### 4.2 ¥0 になったのにアカウントが作成されない

**確認順序**:

1. **Stripe で取引が成立しているか**
   ```
   Stripe Dashboard → 取引 → 該当メアドで検索
   ```
   `¥0` の取引が「成功」になっているか確認。

2. **Webhook が成功しているか**
   ```
   Stripe Dashboard → 開発者 → Webhooks → お豆奏法 Stripe Webhook 本番
   ```
   該当時刻の event が HTTP 200 で完了しているか確認。

3. **auth.users / public.users が作成されているか**(SQL)
   ```sql
   select 
     au.email,
     au.created_at,
     case when pu.id is null then '❌ 欠落' else '✅ OK' end as status
   from auth.users au
   left join public.users pu on au.id = pu.id
   where au.email = '<該当メアド>';
   ```

詳細な調査手順は `docs/2026-06-25_incident_webhook.md` §5 を参照。

### 4.3 招待者にメールが届かない

**原因の候補**:
- 迷惑メールフォルダに入っている
- キャリアメール(docomo, au, softbank, icloud)が Supabase Auth メールをブロック
- メールアドレスのタイプミス

**対応**:
- 迷惑メールフォルダ確認を案内
- 別のメアド(Gmail 等)での再登録を打診
- 既に auth.users が作成されていれば、Supabase Dashboard から手動でパスワードリセットメールを送信

### 4.4 同じ人が複数回使ってしまった

ERINA-GIFT は「1ユーザーあたり1回」設定のため、同じメアドでは2回目以降使えない。
ただし別メアドで再使用は可能。
意図的な多重利用を防ぎたい場合は、Stripe Dashboard で 1 ユーザーあたりの上限を厳密管理。

## 5. 設計判断と背景

### 5.1 なぜ独自実装ではなく Stripe 標準機能を採用したか

候補は3案あった:

| 案 | 採用 | 理由 |
|---|---|---|
| Stripe Coupon + Promotion Code | ✅ 採用 | 取引履歴に残る・Dashboard で完結・実装が1行 |
| 独自実装(Checkout スキップ) | ❌ | Stripe 取引履歴に残らず可視性が低い |
| 手動 role 付与 | ❌ | 完全に属人化、ヒューマンエラーのリスク大 |

### 5.2 なぜ共通コード(個別コードでなく)を採用したか

10名規模なら共通コード `ERINA-GIFT` で十分。個別コードを発行すると:
- Stripe Dashboard の管理が煩雑
- 「コードが流出した」際の対応コスト

10名規模で SNS 流出のリスクは、えりな先生から個別 LINE で送る運用で抑制可能と判断。

### 5.3 なぜ ¥0 取引でアフィリエイト報酬を発生させないか

紹介経由で ¥0 ギフト購入が起きた場合、紹介者に報酬が発生すると:
- 実質的に「報酬目当ての無料配布乱用」が可能になる
- えりな先生の招待意図と乖離する

報酬計算が `session.amount_total` を基にしているため、¥0 取引は自動でスキップされる。これは意図的に維持する設計。
