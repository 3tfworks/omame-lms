### 2.2 ¥0 取引時の挙動(新規仕様)

ギフトクーポン適用で `session.amount_total === 0` の場合、報酬計算は **自動でスキップ** される(`amount_total` を基に報酬を算出する既存ロジックの自然な帰結)。

- 紹介経由でギフト購入された場合も報酬は発生しない(意図的な挙動)
- これは設計上正しく、変更不要

## 6. 重要な設計判断と根拠(本セッション分)

### 6.1 マーケ判断

#### 判断: ギフトクーポンは「えりな先生から特別招待」の文脈で配布する

- **背景**: 障害のお詫び + えりな先生の人脈を活かした拡販
- **採用**: Stripe ¥0 クーポン + 共通プロモコード `ERINA-GIFT`
- **却下**: 個別コード発行(運用コスト過大)
- **理由**: 10名規模であれば共通コードで十分。SNS 等への流出リスクは「えりな先生から個別 LINE で送る」運用で抑制

#### 判断: 問い合わせ窓口を3箇所に設置

- **背景**: 障害発生時、購入者が連絡手段に迷っていた
- **採用**: LP FAQ / LMS サイドバー / ログイン画面の3箇所に LINE 窓口
- **理由**: 「困った時の流入経路」を複数確保することで、同種障害の検知を早める

### 6.2 技術判断

#### 判断: webhook ハンドラを冪等化(再送で安全な設計)

- **背景**: 障害復旧時、過去 webhook を全件再送する必要があった
- **採用**:
  - public.users → upsert(onConflict: 'id')
  - affiliate_rewards → upsert(onConflict: 'stripe_event_id')+ 新規ユニーク制約
  - listUsers → 全件ページネーション
- **却下**: 「再送せず手動でレコード作成」案
- **理由**:
  - 13名の手動レコード作成はミスのリスク大
  - 将来も同種障害は起こり得るため、再送で復旧できる仕組みが必要
  - 冪等化は webhook の「定石」であり、技術的負債を解消する好機

#### 判断: Migration → コードデプロイの順序を厳守

- **背景**: 新カラムを使うコードを先にデプロイすると、Migration 未実行の間エラーが連発する
- **採用**: Supabase Dashboard SQL Editor で Migration 先行実行 → `vercel --prod`
- **理由**: ダウンタイムを発生させない安全な順序

#### 判断: ギフトは Stripe 標準機能(クーポン + プロモコード)で実現

- **背景**: 「無料配布」の選択肢として、独自実装 / Stripe 機能 / 手動 role 付与の3案
- **採用**: Stripe Coupon(100%off)+ Promotion Code + `allow_promotion_codes: true`
- **却下**:
  - 独自実装(コードで Checkout をスキップして role 直書き)→ Stripe 取引履歴に残らず可視性が低い
  - 手動 role 付与 → 完全に属人化
- **理由**: Stripe Dashboard で発行・期限管理・引き換え回数管理が完結。コード変更は1行のみ(`allow_promotion_codes: true`)
